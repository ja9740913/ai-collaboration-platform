# 案例研究｜四週 AI 客服試點提案評審

## 問題

提案要求在四週內導入 AI 客服助理。單一回答很容易只談模型效果，忽略個資、轉人工、測試資料、責任人與驗收方式。

公開工作流將同一份需求交給三個責任不同的角色：

- `risk`：資料、權限、錯誤回答及責任風險。
- `delivery`：時程、範圍、依賴與可驗收交付。
- `user`：使用者理解、信任、轉人工與操作摩擦。

## 輸入

輸入位於[`fixtures/customer-service-case.json`](../fixtures/customer-service-case.json)，包含問題、三項限制與三項成功指標。成功指標是試點的「目標」，不是已達成的商業成果。

## 故意注入的失敗

風險角色第一次執行會回傳匿名 Fixture `RATE_LIMITED`。Retry Policy 會：

1. 確認錯誤可重試。
2. 取政策退避與 `retryAfterMs` 的較大值。
3. 發出 `retry_scheduled` 事件。
4. 重新執行風險角色。

這使公開案例能實際驗證錯誤路徑，而不只在文件裡宣稱「支援重試」。

## 輸出

三個角色的結構化輸出會被驗證後合成：

- 建議：`pilot_with_gates`。
- 必要控制：低信心轉人工、個資遮罩、固定 golden set、轉接保留上下文。
- 驗收測試：回答可接受率、P95 延遲、錯誤回答轉人工、權限與 Schema 測試。
- 分歧：交付角色支持試點；風險與使用者角色要求附帶條件。

## 驗證證據

`npm run check` 目前涵蓋 11 個測試：

- 物件鍵順序不影響 `inputHash`。
- 語意輸入改變會改變 Hash。
- 暫時錯誤依政策重試。
- 永久錯誤只執行一次。
- 合法 Schema 通過。
- 錯誤角色與 Severity 被拒絕。
- 三角色工作流完成並產生決策。
- 完成的工作流可由 Checkpoint 續跑且不再呼叫 Provider。
- Hash 不一致時拒絕續跑。
- FileRunStore 可跨實例讀回 Checkpoint。
- FileRunStore 拒絕路徑穿越。

## 尚未宣稱的成果

本案例沒有宣稱已替真實客服降低工時、提高解決率或達成上述成功指標。商業成效需要正式試點與基準資料後才能報告。
