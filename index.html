01 / Runnable case

一個案例做深：客服 AI 該直接上線，還是先做受限試點？

按下執行會重播與公開 Fixture 相同的事件。這不是連到付費模型的表演，而是能用 npm run demo 在本機重現的確定性案例。

匿名化提案輸入

customer-service-case.json 決策問題

是否在四週內把 AI 客服導入正式服務流程？

限制條件
  • 先從內部 FAQ 與低風險問題開始
  • 敏感資料不得送入未核准的第三方模型
  • 低信心回答必須能帶上下文轉真人
成功標準
  • 50 題 golden set 可接受率達 90%
  • P95 回覆時間低於 8 秒
  • 錯誤回答轉人工率 100%

事件追蹤

固定輸入、固定回應、可重現錯誤路徑

等待執行。流程將並行啟動 risk、delivery、user 三個角色。

Structured decision pilot_with_gates

先做四週受限試點；加入資料遮罩、低信心轉人工、golden set 驗收與完整稽核軌跡，再決定是否擴大。

瀏覽器只重播公開事件，真正的編排、驗證、重試與恢復邏輯位於 TypeScript 原始碼。查看 CLI 入口、查看 Fixture。

02 / Architecture

從輸入到決策，每一層都有清楚的失敗邊界

編排器不信任模型輸出。它負責輸入雜湊、角色路由、重試策略、Schema 驗證、狀態保存與決策彙整。

Input boundary

提案＋限制＋驗收指標

穩定序列化後計算 inputHash，避免用錯輸入恢復舊任務。

→
Orchestrator

平行路由與錯誤治理

riskdeliveryuser

限流／暫時錯誤可重試；不合法輸出不能混入最終決策。

→
Output boundary

決策＋Controls＋Tests

Checkpoint 保存角色結果與事件，失敗後可從已完成的角色繼續。

Deterministic hash相同輸入產生相同識別；不同輸入拒絕錯誤恢復。
Typed failure錯誤碼與 retryable 明確分離，不靠字串猜測。
Atomic checkpoint檔案先寫暫存再 rename，降低中斷造成半份狀態。

04 / Honest ownership

AI 是協作者；架構判斷與驗收責任由我承擔

公開作品不把「用了很多 AI」包裝成個人全手寫。我負責定義問題、邊界、設計與驗收；AI 用於加速探索與產生候選實作。

我的責任

  • 把模糊的展示需求改寫成可驗證的工程問題
  • 定義工作流邊界、資料合約、錯誤分類與恢復策略
  • 設計 Fixture、測試情境、驗收條件與安全公開範圍
  • 逐項審查、執行測試並對最後發布結果負責

AI 協作範圍

  • 協助列舉失敗情境與測試案例
  • 產生候選程式、文件與介面文案
  • 提供 reviewer 視角與反例

更完整的分工、來源邊界與驗證方式記錄在 ownership 文件。

05 / Limits & security

公開作品刻意縮小範圍，也清楚標示沒有證明什麼

◫

匿名 Fixture

不含真實客戶資料、提示詞、模型金鑰或公司內部程式。公開版證明編排方法，不公開私人系統。

⌁

非壓力測試

目前證明功能與錯誤路徑，不代表已通過大流量、跨區容錯或正式 SLA。

↗

可替換 Provider

公開版使用確定性 Provider；接真實模型需自行加入金鑰治理、成本限制與內容政策。
